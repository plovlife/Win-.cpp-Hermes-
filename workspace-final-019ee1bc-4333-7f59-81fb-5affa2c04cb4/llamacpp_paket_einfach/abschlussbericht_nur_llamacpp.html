Abschlussbericht und Tutorial

Aufbau eines lokalen Hermes-Agent-Backends ausschließlich mit llama.cpp unter Windows 11

Kernaussage: Für den vorliegenden Anwendungsfall ist ein direktes llama.cpp-Setup mit langem Kontext, einem Hermes-tauglichen 8B-GGUF-Modell und einer einfachen Windows-Startdatei die sinnvollste Lösung.

1. Ziel des Projekts

Dieses Projekt verfolgt das Ziel, einen vollständig lokalen Modellserver bereitzustellen, der als Backend für Hermes Agent dient. Der Server soll OpenAI-kompatibel erreichbar sein, lange Kontexte unterstützen, auf Consumer-Hardware stabil laufen und zugleich so dokumentiert werden, dass er sich für spätere wissenschaftliche Auswertungen eignet.

2. Warum nur llama.cpp?

llama.cpp bietet einen direkten Zugriff auf die Inferenzschicht. Der offizielle llama-server unterstützt OpenAI-kompatible Endpoints, WebUI, Parallel-Slots, Continuous Batching, Function Calling, Monitoring und weitere Funktionen. Dadurch lässt sich eine saubere, reproduzierbare Architektur ohne zusätzliche Abstraktionsschichten aufbauen.

3. Empfohlenes Modell

  • Primär: Hermes-3-Llama-3.1-8B-GGUF
  • Empfohlene Quantisierung: Q6_K oder Q5_K_M

Für Agent-Workflows ist nicht die maximale Modellgröße entscheidend, sondern die Kombination aus Strukturtreue, Tool-Eignung, ausreichendem Kontextfenster und stabiler Laufzeit.

4. Zielkonfiguration

llama-server.exe \
  -m C:\000.mAIn\Model\01_LLMs\GGUF\hermes-3-llama-3.1-8b-q6_k.gguf \
  --host 127.0.0.1 --port 8080 \
  -ngl all \
  -c 65536 \
  -np 1 \
  -ctk q8_0 -ctv q8_0 \
  -fa on \
  -b 2048 -ub 512 \
  -t 8 \
  --jinja

5. Bedeutung der wichtigsten Parameter

  • -m: Pfad zur Modell-Datei
  • --host / --port: lokale Erreichbarkeit des Servers
  • -ngl all: moeglichst viel Modell auf GPU
  • -c 65536: langes Kontextfenster fuer Hermes
  • -np 1: ein voller Slot statt geteilter Parallelisierung
  • -ctk / -ctv q8_0: kompakterer KV-Cache
  • -fa on: Flash Attention aktiv
  • --jinja: saubere Chat-Template-Verarbeitung

6. Hermes-Anbindung

7. Vereinfachter Alltagseinsatz

Im beigefuegten Paket befinden sich bereits fertige Dateien fuer den praktischen Betrieb:

  • START_llama_server_GUI.bat – startet den Server per Doppelklick
  • START_llama_server_GUI.ps1 – fragt beim ersten Start nach llama-server.exe und dem GGUF-Modell
  • STOP_llama_server.bat – stoppt laufende Instanzen
  • SERVER_TEST_UND_WEBUI.bat – oeffnet WebUI und testet den Endpoint

9. Fazit

Ein ausschließlich auf llama.cpp basierender Aufbau ist für dieses Projekt die fachlich sauberste und praktisch sinnvollste Lösung. In Kombination mit einem Hermes-tauglichen 8B-Modell, langem Kontext und einem einfachen Windows-Startmechanismus entsteht ein lokales System, das sowohl im Alltag als auch als Ausgangspunkt für technisch strukturierte oder akademisch orientierte Arbeiten tragfähig ist.
